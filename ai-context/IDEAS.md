## Stage 1 калибровка: examples + feedback — переосмысление (брейншторм 2026-05-31)

### Как сейчас (для контекста)

`stage1_examples` (admin-curated) + feedback (из голосов юзеров) рендерятся в плоский текст и
вклеиваются в system-промпт Stage 1: `SYSTEM_PROMPT + examplesContext + feedbackContext`
(см. `src/analysis/scorer.js` → `_buildExamplesContext` / `_buildFeedbackContext`).
Это few-shot калибровка **шкалы вообще**, одинаковая для всех трендов в батче, soft, не fine-tuning.
Каждая запись — голый ярлык: `"title" [category] → memePotential N — rationale`.

### Проблема

На **тяжёлых** нарративах калибровка либо бесполезна, либо мешает. «Тяжёлые» = все три типа сразу:
1. **Нужен контекст** — смысл зависит от предыстории/отсылки, которой нет в данных.
2. **Неочевидный потенциал** — серьёзное на вид событие, где надо вдуматься, родится ли мем (легко отмахнуться шаблоном «политика→0»).
3. **Обманчивый title** — заголовок скучный, но в контенте сильный нарратив.

Два ключевых инсайта из брейншторма:
- **Асимметрия репрезентации.** На ВХОД Stage 1 про тренд идёт много (title + description + PreStage/Gemini:
  topic, entities, slang, visual/video/audio/speech, mood, authoritative GeminiScore — ~200-500 токенов).
  А калибровочные примеры — голые ярлыки (~30 токенов). Модель оценивает богато описанный тренд,
  сверяясь с бедными ярлыками. Примеры учат маппингу `вход→число`, но НЕ учат КАК РАССУЖДАТЬ.
  Для лёгких кейсов маппинга хватает; для тяжёлых нужен образец мышления / больше данных — а их нет.
- **Замкнутый круг Stage 2.** Глубокий путь (Stage 2, x_search) гейтится по `memePotential >= threshold`
  из Stage 1. Тренд с обманчивым title Stage 1 занижает → он не проходит порог → не получает x_search →
  второго шанса нет. Самые тяжёлые кейсы выпадают из глубокого анализа именно потому, что Stage 1 их недооценил.

Constraint: «отдать весь контекст всем» нельзя — сотни трендов/цикл, батчи по 8, затраты взлетят.

### Вывод-разворот

Возможно, правильно не «улучшать examples/feedback», а **разгрузить их от того, на что они не способны**:
их работа — калибровать лёгкую массу (это норм), а тяжёлые — задача отдельного механизма (см. B).

### 🟢 B — Двухскоростная оценка [✅ SHIPPED 2026-05-31]

Stage 1 остаётся дешёвым для всех. Поверх — детектор «тяжёлых», эскалирующий подозрительные
(в т.ч. **недооценённые**) на глубокий путь, **независимо от meme-score**, под жёстким потолком стоимости.

- Тратим дорогое только на тяжёлые → уважает constraint.
- Чинит замкнутый круг Stage 2 (недооценённый тяжёлый получает второй шанс).
- Покрывает все 3 типа разом.

> **Дизайн-спека:** `docs/superpowers/specs/2026-05-31-stage1-deep-escalation-design.md`.
> Принятые решения:
> - **Детекция = комбо:** эвристика (код) «объективная активность ↑ при memePotential ↓ и низком junk»
>   (поля `emergenceScore` / `metrics.twitter.viralityScore` / `clusterMetrics.itemCount` / `junkPenalty`)
>   **+** флаг модели `needsDeeperLook` в `STAGE1_RESPONSE_SCHEMA`. Любой сработал → эскалация.
> - **Глубокий путь = переиспользуем Stage 2** `_stage2DeepDive` (x_search), его НЕ меняем.
> - **Reasoning-слой за тогглом** `deepReasoningEnabled` (дефолт OFF) = смена модели Грока на reasoning-вариант
>   (не игнорируемый xAI param!), а не отдельный вызов. Включим, когда подтвердим цену.
> - **Бюджет = общий пул** `stage2MaxCalls` + резерв `escalationReserve` слотов под эскалации, с reflow.
> - google_trends (без URL) не эскалируем — x_search их не копнёт.
> - Открытые детали (пороги эвристики, имя reasoning-модели) → в плане. Дальше: writing-plans.

### 🟡 A — Worked examples: ярлык → разбор [после B]

Превратить `stage1_examples` из ярлыков в few-shot с **полным рассуждением**:
«вот кейс → как думаю шаг за шагом → почему такой балл». Учим модель процессу мышления, а не маппингу.

- Дёшево, кэшируется в префиксе, пайплайн не трогаем.
- Бьёт в «нет образца рассуждения» (тип 2 — неочевидный потенциал).
- Минусы: слабо помогает типам 1 и 3 (там не хватает ДАННЫХ, а не рассуждения); каждый пример пухнет ×5-10;
  всё равно общий, не адаптируется под конкретный тренд.
- Хорошо ложится сверху на B как «топливо» для reasoning-режима на эскалированных трендах.

### 🔵 C — Дистилляция feedback + examples в «правила» [будущее]

Раз в N часов мета-LLM сжимает накопленные examples + голоса в короткий свод принципов,
вместо сырых ярлыков. Feedback перестаёт быть списком лайков — становится выученными правилами.

- Лечит дрейф/шум feedback (накрутка лайков, мусорные причины), компактнее префикс, выше сигнал-на-токен.
- Отдельная история про гигиену калибровки — НЕ про тяжёлые кейсы. Тяжёлым прямо не поможет.

#### ❌ Grok Build CLI по подписке вместо API — ПРОВЕРЕНО, НЕ РАБОТАЕТ (2026-06-01)

Гипотеза оператора: гонять скоринг через `grok` CLI (Grok Build) по подписке SuperGrok ($30/мес)
вместо xAI API ($6-10/день). Развёрнута песочница (Vultr 185.192.23.55), CLI v0.2.14 поставлен,
device-auth по подписке (`grok login --device-auth` → OAuth, `auth_mode: oidc`, БЕЗ API-ключа) — **работает**.
Прогнали реальные тесты, не доверяя ни докам, ни словам самой модели (тык + streaming-json + чтение исходников MCP).

Итог по трём путям:
- **Stage 1 через CLI** — ✅ технически работает (валидный `{"trends":[...]}` на реальном прод-промпте из 12 трендов,
  качество скоринга адекватное), **НО латенси 78с/батч**. На сотнях трендов/цикл (`ceil(N/8)` батчей) пайплайн
  не успевает закрыть цикл + упрётся в throttling SuperGrok (~100 промптов/2ч). Экономия мизерная — Stage 1 и так дёшев.
- **Stage 2 (x_search) встроенным тулом CLI** — ❌ **тула X-поиска в CLI НЕТ**. Built-in только `web_search`/`web_fetch`/
  `bash`/`read_file`/… (подтверждено on-disk доками CLI + тыком через `--tools x_search`: модель гадала имена
  `x_keyword_search`/`search_x`, зависла на «stand by for tool results», реального `tool_use`-события в streaming-json НЕ было).
- **Stage 2 через MCP** — ❌ оба open-source X-search MCP-сервера (`toocheap/x-search-mcp`, `stat-guy/grok-search-mcp`)
  внутри ходят в xAI **по `XAI_API_KEY`** (per-token) либо по платному X API v2. Цитата из доки: «A SuperGrok subscription
  alone is insufficient». MCP — лишний слой поверх того же платного канала, экономия = 0.

**Фундаментальная причина:** то, что жжёт деньги (x_search заливает ~1K токенов/твит обратно в контекст,
~12K input/вызов), физически тарифицируется по API в ЛЮБОМ канале. Подписка даёт интерактивный «мозг», но не
бесплатный программный x_search. **Идею закрыли.** Песочницу можно гасить (OAuth-токен оператора подчищен `grok logout`).

**Что реально срежет счёт (НЕ сделано, ждёт решения оператора — трогает качество скоринга, нужно явное добро):**
