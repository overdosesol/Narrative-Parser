# AGENT RULES (для любых моделей)

Этот файл обязателен для всех агентных сессий (GPT/Codex, Claude, Gemini и др.).

## 1) Перед началом работы

1. Прочитай `ai-context/SESSION_CONTEXT.md`.
2. Быстро проверь актуальное состояние репозитория (`git status`, ключевые файлы задачи).

## 2) Во время работы

1. Не ломай существующий flow без явной причины.
2. Если меняешь архитектуру/контракты API — зафиксируй это в контексте.
3. Если есть риск/компромисс — явно опиши его в `WORKLOG.md`.
4. Всегда отвечай пользователю на русском языке, сленг и англицизмы разрешены. Общайся на ТЫ.
5. Не скидывай пользователю код в чат и старайся объяснять по проще

## 3) После выполнения (обязательно)

1. Обнови `ai-context/WORKLOG.md` (что сделано, какие файлы, результат).
2. Обнови `ai-context/SESSION_CONTEXT.md` (текущее состояние проекта).

## 4) Формат записи в WORKLOG

- Дата/время
- Модель/сессия
- Цель
- Изменения (по файлам)
- Деплой/проверка
- Риски/заметки

## 5) Запреты

- Не удалять файлы из `ai-context/` без явной просьбы владельца.
- Не оставлять задачу «в подвешенном состоянии» без заметки в `WORKLOG.md`.
- Не выполнять destructive git-команды (`reset --hard`, `checkout --`, force push) без явного запроса владельца.

## 6) Ротация WORKLOG → ARCHIVE

`WORKLOG.md` держим как rolling window последних **~10 entries**. Когда после
свежей записи количество `## 2026-...` заголовков становится **больше 12**:

1. Найди границу top-10 (10 новейших entries сверху файла).
2. Перенеси все entries ниже в начало `WORKLOG_ARCHIVE.md` (новейшие
   архивированные сверху, старейшие снизу — порядок сохраняем).
3. Граница entry — это блок от `## 2026-...` до строки `---` перед следующим
   `## 2026-...` (или EOF). Хедер `# WORKLOG ARCHIVE` в архиве не трогаем.
4. После перемещения ничего не правь содержательно — только перенос.

Зачем: WORKLOG читают LLM-агенты при старте сессии. Каждые ~50K токенов файла =
~$0.15 за чтение без кэша + ~25% контекстного окна. Rolling window держит
файл компактным без потери истории (архив + git сохраняют всё).

**Не делай ротацию** если:
- Свежая запись пользователя ещё «горячая» (пользователь только что просил поправить контекст).
- Если активные 10 entries относятся к одному большому PR — можно временно держать до 12.
- Если пользователь явно просил оставить конкретную старую запись на виду.

## 7) SESSION_CONTEXT — state, не change

`SESSION_CONTEXT.md` описывает **текущее состояние проекта**, не историю изменений.
Историю пишем в `WORKLOG.md`.

**Запрещено в SESSION_CONTEXT**:
- Заголовки с датами вида `### Название (2026-XX-XX)` — это маркер ship-log entry, такому место в WORKLOG.
- Прозы вида «раньше было X, переделали в Y, теперь работает Z» — оставь только описание Z.
- Tombstones удалённых фич («Персонализированный ранг — УДАЛЁН ...») — упомяни одной строкой в Files map / Env / соответствующей секции, или вообще не упоминай.
- TODO-заметки («поправить env на проде», «рассмотреть удаление nano») — это в WORKLOG или issue tracker.

**Что должно быть в SESSION_CONTEXT**:
- Бизнес-правила (планы, лимиты)
- Архитектурные диаграммы (pipeline order)
- Текущие defaults / пороги / формулы
- Карта файлов (1-line per module)
- Critical traps / gotchas (вроде «Ловушка server.js»)
- Env keys минимальный набор
- Cross-references между модулями (что-куда читает)

**Тест на принадлежность секции**: представь что новый агент через месяц читает только SESSION_CONTEXT и работает с проектом. Если без этой секции он не справится — это state, оставь. Если справится (потому что фича уже в коде, а текст лишь описывает «как мы её делали») — это change, перенеси в WORKLOG.

**Сжимай безжалостно**. Цель — описывать сейчас, не вспоминать как пришли. Бенчмарк размера: <12K токенов (~500 строк). Превышение — повод трим-пасса.
